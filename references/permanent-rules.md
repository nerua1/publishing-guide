# Permanent Publishing Rules — Lessons From Failed Sessions

## Rule 1: NEVER Fragments
When fixing a file, prompt, or command — give the ENTIRE thing from scratch. 
No "change line X to Y". No "add after line Z". One complete output.

**Why**: User gets frustrated assembling pieces. They want copy-paste-ready results.

## Rule 2: ONE Command Max
When giving a command to run manually — maximum ONE. Chain everything. 
No "first do this, then that, then..."

**Why**: Multiple commands create friction and errors. One chain is executable immediately.

## Rule 3: Check Rules Before Publishing
Always read the publishing-guide and skill-vetter BEFORE any ClawHub/GitHub publication. 
Anti-duplicate check is mandatory. Never publish a skill that already exists with better scores.

**Why**: We published 5 skills without checking — ralph and memory-palace had to be deleted because they were worse duplicates.

## Rule 4: Run Code As-Is First
When given existing working code (like o net.txt), run it FIRST. Don't convert, don't refactor, don't migrate to a different framework. Show the user what they have before changing anything.

**Why**: We spent hours converting a working Vite project to Next.js, producing a broken mess. The original code was 9.5/10 design.

## Rule 5: Cover Images Always
Every Dev.to article and GitHub repo MUST have a cover image (1280×640px). 
Generate the AI prompt BEFORE writing the article. Never publish without an image.

**Why**: Tomek explicitly requires this. Published articles without images waste the publication.
